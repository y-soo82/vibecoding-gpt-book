# 바이브코딩(vibe coding) 1차 GPT(Generative Pre-trained Transformer) 입문서 출판 계획

## 출판 방향

이 책은 ChatGPT 대화창만으로 첫 웹사이트를 만들고, GitHub 연동을 통해 웹에서 열리는 결과물을 공개하는 과정을 다룹니다. 독자는 개발자가 아니라 웹서핑과 유튜브 정도에 익숙한 일반 사용자입니다.

이번 MVP는 1차 GPT(Generative Pre-trained Transformer) 입문서에만 집중합니다. 2차 Codex 실습서는 별도 책으로 분리하며, 이 책에서는 Codex App을 본격 설명하지 않습니다.

디자인 기준은 `디자인_가이드.md`를 따릅니다. 전체 톤은 화이트 기반의 깔끔한 실습형 반응형 전자책이며, Chapter 2와 Chapter 3은 설명, 프롬프트(prompt) 박스, HTML(HyperText Markup Language) 결과 이미지 흐름을 반복합니다.

## 책 제목 후보

1. 보면서 따라 하는 바이브코딩(vibe coding) 입문
2. ChatGPT로 처음 만드는 내 웹사이트
3. 그림 한 장과 한 줄 부탁으로 시작하는 바이브코딩(vibe coding)

## 타겟 독자

- ChatGPT는 들어 봤지만 대화창에서 무엇을 부탁해야 할지 자주 멈추는 독자입니다.
- GitHub, 배포, 저장소, Pages 같은 단어가 낯선 독자입니다.
- 코드를 읽거나 파일을 직접 만들고 싶지 않은 독자입니다.
- 컴퓨터보다 휴대전화 화면이 더 익숙할 수 있습니다.
- 영어 UI(User Interface), 권한 허용, 공개 배포 단계에서 불안을 느낄 수 있습니다.

## 책의 핵심 약속

- 독자는 ChatGPT 대화창 안에서 한 줄씩 부탁합니다.
- 책은 결과를 실제 브라우저(browser) 화면과 모바일(mobile) 화면으로 보여줍니다.
- 독자는 웹에서 열리는 사이트 2개를 완성합니다.
- 코드 작성, 터미널 실행, 로컬 편집기 사용을 독자에게 요구하지 않습니다.
- 막히는 지점마다 지금 눌러야 할 자리와 건드리지 않아도 되는 자리를 분리해서 안내합니다.

## MVP 출판 범위

- 반응형 웹북(web book) 1권을 우선 완성합니다.
- 전체 분량은 80~100페이지 기준으로 설계합니다.
- 이미지는 현재 50장 기준으로 구성합니다.
- 중복 제거를 우선하므로 이미지를 60~70장으로 억지로 채우지 않습니다.
- Chapter 1에는 결과물 이미지를 넣지 않습니다.
- 같은 이미지 파일은 책 전체에서 한 번만 사용합니다.
- 결과 사이트는 2개를 목표로 합니다.
  - 첫 실습 사이트: 이미지 한 장과 한 줄 부탁으로 만든 반응형 웹사이트
  - 본격 실습 사이트: 타로(tarot) 카드 자산과 리딩 흐름을 가진 반응형 사이트
- GitHub Pages 배포는 책의 핵심 경험에 포함합니다.
- 카카오 공유 같은 외부 서비스 등록 구간은 MVP에서 제외하고, 부록에서 후속 확장으로 안내합니다.

## 후속 EPUB(Electronic Publication)/PDF(Portable Document Format) 전환 범위

- MVP 웹북(web book) 검수 후 EPUB(Electronic Publication) 전환을 진행합니다.
- EPUB(Electronic Publication) 전환 시에는 이미지 경로, 표지, 목차, 글꼴 라이선스, 리더기 호환 CSS(Cascading Style Sheets)를 별도로 점검합니다.
- PDF(Portable Document Format)는 웹북(web book)과 EPUB(Electronic Publication)의 문장·이미지 검수가 끝난 뒤 마지막 산출물로 분리합니다.
- EPUB(Electronic Publication)/PDF(Portable Document Format) 단계에서는 원고 내용을 새로 확장하지 않고, 포맷별 읽기 품질만 조정합니다.

## 재구성 방식

Claude 쪽 원본 repo는 README만 남도록 초기화되어 있어 이전 원고, 이미지, 스크립트를 복구하지 않습니다. 재구성은 다음 자료를 기준으로 합니다.

- Codex e-book 제작 가이드
- vibecoding 프로젝트 규칙
- e-book 에이전트 역할 문서
- Claude task/log에 남은 이전 구조
- 공개 URL(Uniform Resource Locator)과 repo 설명에 남은 책 방향

Claude task/log에서 반영할 구조는 다음과 같습니다.

- 타로(tarot) 사이트는 `home`, `ask`, `spread`, `result` 네 화면 감각으로 설명합니다.
- 리딩 결과는 메이저 22장, 5영역, 정방향과 역방향 개념을 갖습니다.
- 결과 저장은 오늘 결과와 최근 이력 5건 정도로 제한합니다.
- GitHub Pages 배포 흐름은 2부와 3부에 각각 연결합니다.
- 카카오 섹션은 실제 MVP 작업에서 제외하고, 독자에게 지금 따라 할 범위가 아니라고 설명합니다.
- 실제 카드 예시는 The Fool, The Magician, The Star, 통일 스타일 비교, 들쭉날쭉 비교, 모바일(mobile) 결과 화면을 중심으로 재구성합니다.

## 역할 분담

| 역할 | 책임 | 산출물 |
|---|---|---|
| Planner | 목차, 작업 단위, 검수 반영 범위 결정 | 목차, 작업 단위표, 재작업 지시 |
| Writer | Planner 명세에 따라 한 단위씩 본문 작성 | 섹션 원고 |
| Editor | 문장, 톤, 가독성 정리 | 교정본 |
| Reviewer | 타겟 독자 시점으로 이탈 위험 보고 | 페이지별 독자 반응 |
| Verifier | 금지 기준, 이미지, 링크, 구조 검증 | 결함 목록과 판정 |
| Capture | ChatGPT, GitHub, 결과 사이트 화면 캡처(capture) | 출판용 PNG 목록 |
| EPUB(Electronic Publication) Builder | 웹북(web book) 검수 후 EPUB(Electronic Publication) 패키징 | EPUB(Electronic Publication), 검증 로그 |

## 금지 기준

- 깨진 이미지가 독자에게 보이면 안 됩니다.
- 같은 이미지가 두 챕터 이상 반복되면 안 됩니다.
- Chapter 1에서 첫 웹사이트 결과나 타로(tarot) 사이트 결과를 선공개하지 않습니다.
- 본문 프롬프트(prompt) 박스와 같은 내용을 이미지로 다시 보여주지 않습니다.
- 독자에게 직접 파일을 만들거나 확장자를 바꾸라고 안내하지 않습니다.
- 독자에게 터미널, CLI, 로컬 편집기를 사용하라고 안내하지 않습니다.
- ChatGPT 대화창 안 미리보기 화면을 결과물로 캡처(capture)하지 않습니다.
- 실제 결과물은 ChatGPT 밖 브라우저(browser) 화면 또는 배포 URL(Uniform Resource Locator)에서 캡처(capture)합니다.
- 자체 SVG 대체 이미지를 카드나 결과물처럼 쓰지 않습니다.
- 개인정보가 보이는 캡처(capture)를 출판본에 넣지 않습니다.
