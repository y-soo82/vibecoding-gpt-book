# 바이브코딩(vibe coding) 1차 GPT(Generative Pre-trained Transformer) 입문서 목차 v1

## 전체 기준

- 목표 독자: 웹서핑과 유튜브 시청 정도에 익숙한 일반 독자
- 목표 경험: 책만 따라 하면 웹에서 열리는 타로(tarot) 사이트를 완성합니다.
- 목표 결과: 기능, 화면 흐름, UI(User Interface), UX(User Experience)가 책에 실리는 HTML(HyperText Markup Language) 캡처(capture)와 최대한 일치해야 합니다.
- 출판 형태: 이미지 70%, 글 30% 기준의 반응형 웹 전자책
- 최종 전환: Sigil 기반 EPUB(Electronic Publication) 출판을 염두에 둡니다.
- 핵심 방식: 설명, 프롬프트(prompt), 실제 HTML(HyperText Markup Language) 결과 캡처(capture)를 반복합니다.
- 결과 검증: HTML(HyperText Markup Language) 결과는 Playwright로 ChatGPT 입력 화면과 실제 결과 화면을 캡처(capture)합니다.

## Chapter 1. AI(Artificial Intelligence), LLM(Large Language Model), 바이브코딩(vibe coding) 개요와 시장 소개

예상 분량: 8~12페이지  
예상 이미지: 0장

### 목적

독자가 왜 지금 바이브코딩(vibe coding)을 배워야 하는지 이해하게 합니다. ChatGPT 화면 사용법으로 바로 들어가기 전에, AI(Artificial Intelligence)와 LLM(Large Language Model)의 흐름, 바이브코딩(vibe coding)의 의미, 일반 독자에게 열리는 제작 가능성을 먼저 설명합니다.

### 독자 행동 목표

- AI(Artificial Intelligence)와 LLM(Large Language Model)을 지나치게 어렵게 느끼지 않습니다.
- 바이브코딩(vibe coding)이 “코드를 외우는 일”이 아니라 “AI(Artificial Intelligence)에게 결과를 부탁하고 고치는 방식”임을 이해합니다.
- 이 책의 목표가 타로(tarot) 사이트 결과물 완성이라는 점을 압니다.
- LLM(Large Language Model) 결과가 매번 완전히 같지는 않아도, 기능과 화면 목표를 맞춰 가는 책이라는 점을 이해합니다.
- 결과 이미지는 Chapter 2와 Chapter 3에서 실제 장면으로 한 번만 본다는 원칙을 이해합니다.

### 구성

- 1-0. 이 책은 무엇을 만드는 책인가요
- 1-1. AI(Artificial Intelligence)와 LLM(Large Language Model)을 생활 속 도구로 이해합니다
- 1-2. ChatGPT가 결과물을 만드는 흐름을 봅니다
- 1-3. 바이브코딩(vibe coding)은 무엇이 다른가요
- 1-4. 왜 지금 바이브코딩(vibe coding) 입문이 필요한가요
- 1-5. 이제 첫 실습으로 넘어갑니다

### 이미지 방향

- Chapter 1에는 결과물 이미지와 반복 설명 이미지를 넣지 않습니다.
- 첫 웹사이트 결과 이미지는 Chapter 2에서만 사용합니다.
- 타로(tarot) 사이트 결과 이미지는 Chapter 3에서만 사용합니다.
- 개념 설명이 필요할 때도 이미지 대신 짧은 문단과 다음 장 연결 문장으로 처리합니다.

## Chapter 2. 이미지 하나로 웹사이트 만들고 배포합니다

예상 분량: 25~30페이지  
예상 이미지: 14장

### 목적

독자가 이미지 하나를 ChatGPT에 넣고 프롬프트(prompt)를 보내 HTML(HyperText Markup Language) 결과를 받은 뒤, Git 연결과 배포 URL(Uniform Resource Locator) 확인까지 한 번에 경험하게 합니다.

### 독자 행동 목표

- ChatGPT 입력창과 첨부 단추를 찾습니다.
- 이미지 하나를 첨부하고 프롬프트(prompt)를 입력합니다.
- 본문에는 프롬프트(prompt) 핵심 3줄만 보고, 복사하기 버튼으로 전문을 복사하는 구조를 이해합니다.
- ChatGPT 앱에서 Git을 연결합니다.
- 구글(Google) 계정 기준으로 로그인 흐름을 따라갑니다.
- HTML(HyperText Markup Language) 결과를 Git에 푸시하고 배포 URL(Uniform Resource Locator)을 확인합니다.
- 데스크톱(desktop)과 모바일(mobile)에서 배포 URL(Uniform Resource Locator)을 확인합니다.

### 한 장면 기본 구조

각 실습 장면은 아래 순서로 구성합니다.

1. 설명
2. 프롬프트(prompt) 박스
3. HTML(HyperText Markup Language) 결과 이미지

프롬프트(prompt) 박스에는 핵심 3줄만 노출합니다. 전문은 우측 상단 복사하기 버튼으로 복사할 수 있게 구성합니다.

### 구성

- 2-0. 첫 실습에서 만들 웹사이트를 봅니다
- 2-1. ChatGPT 입력창과 첨부 단추를 찾습니다
- 2-2. 이미지 하나를 첨부합니다
- 2-3. 첫 프롬프트(prompt)를 입력합니다
- 2-4. ChatGPT가 만든 HTML(HyperText Markup Language) 결과를 확인합니다
- 2-5. 결과를 실제 브라우저(browser) 화면으로 봅니다
- 2-6. ChatGPT 앱에 Git을 연결합니다
- 2-7. 구글(Google) 계정 기준 로그인 흐름을 확인합니다
- 2-8. Git에 푸시하고 배포 URL(Uniform Resource Locator)을 받습니다
- 2-9. 데스크톱(desktop)과 모바일(mobile)에서 배포 URL(Uniform Resource Locator)을 확인합니다

### 캡처(capture) 방향

- Playwright가 ChatGPT 입력 화면을 캡처(capture)합니다.
- Playwright가 프롬프트(prompt) 입력 후 결과 생성 흐름을 캡처(capture)합니다.
- Playwright가 실제 HTML(HyperText Markup Language) 결과 화면을 데스크톱(desktop)과 모바일(mobile)로 캡처(capture)합니다.
- Git 연결과 구글(Google) 계정 로그인 흐름은 개인정보 제거 기준을 적용합니다.
- 배포 URL(Uniform Resource Locator)은 실제로 열리는 화면을 캡처(capture)합니다.

## Chapter 3. 타로(tarot)앱을 만듭니다

예상 분량: 35~42페이지  
예상 이미지: 32장

### 목적

독자가 책을 따라 타로(tarot) 사이트를 완성하게 합니다. 기능, 화면 흐름, UI(User Interface), UX(User Experience)는 책에 실리는 HTML(HyperText Markup Language) 결과 캡처(capture)와 최대한 일치해야 합니다.

### 독자 행동 목표

- 타로(tarot)앱의 화면 흐름을 이해합니다.
- 프롬프트(prompt)를 단계별로 입력해 같은 구조의 결과를 얻습니다.
- home, ask, spread, result 화면 흐름을 확인합니다.
- 카드와 리딩 결과가 어떤 방식으로 이어지는지 이해합니다.
- 데스크톱(desktop)과 모바일(mobile)에서 같은 사이트를 확인합니다.
- 완성된 타로(tarot)앱을 Git에 푸시하고 배포 URL(Uniform Resource Locator)을 확인합니다.

### 한 장면 기본 구조

각 실습 장면은 아래 순서로 구성합니다.

1. 설명
2. 프롬프트(prompt) 박스
3. HTML(HyperText Markup Language) 결과 이미지

프롬프트(prompt) 박스에는 핵심 3줄만 노출합니다. 전문은 우측 상단 복사하기 버튼으로 복사할 수 있게 구성합니다.

### 구성

- 3-0. 완성할 타로(tarot)앱을 먼저 봅니다
- 3-1. 첫 화면을 만듭니다
- 3-2. 질문 입력 화면을 만듭니다
- 3-3. 카드 펼침 화면을 만듭니다
- 3-4. 결과 화면을 만듭니다
- 3-5. 메이저 22장 카드 구조를 반영합니다
- 3-6. 5영역 리딩 결과를 반영합니다
- 3-7. 정방향과 역방향 결과를 반영합니다
- 3-8. 오늘 결과와 최근 이력을 저장합니다
- 3-9. 모바일(mobile) 화면을 확인합니다
- 3-10. 타로(tarot)앱을 Git에 푸시하고 배포 URL(Uniform Resource Locator)을 확인합니다

### 캡처(capture) 방향

- Playwright가 ChatGPT 입력 화면을 캡처(capture)합니다.
- Playwright가 각 단계의 HTML(HyperText Markup Language) 결과를 캡처(capture)합니다.
- HTML(HyperText Markup Language) 결과 화면은 책에 실리는 UI(User Interface), UX(User Experience) 기준이 됩니다.
- 최종 타로(tarot) 사이트는 데스크톱(desktop)과 모바일(mobile)에서 모두 확인합니다.
- LLM(Large Language Model) 응답이 완전히 같지 않더라도 기능과 화면 목표가 맞는지 검수합니다.

## Chapter 4. 부록

예상 분량: 10~14페이지  
예상 이미지: 4장

### 목적

독자가 중간에 막히기 쉬운 화면과 후속 출판 작업을 정리합니다. 권한 화면, 로그인, 배포 대기, 모바일(mobile) 차이, Sigil 전환 기준을 부록에서 확인하게 합니다.

### 독자 행동 목표

- 구글(Google) 계정 로그인 화면에서 당황하지 않습니다.
- Git 연결과 권한 화면에서 확인할 위치를 압니다.
- 배포 URL(Uniform Resource Locator)이 바로 열리지 않을 때 기다릴 기준을 압니다.
- 모바일(mobile) 화면이 책과 조금 다를 때 확인할 기준을 압니다.
- 웹 전자책이 Sigil을 통해 EPUB(Electronic Publication) 출판 형태로 넘어간다는 흐름을 이해합니다.

### 구성

- 4-0. 화면이 책과 조금 다르게 보일 때
- 4-1. 구글(Google) 계정 로그인 화면에서 볼 것
- 4-2. Git 연결 권한 화면에서 확인할 것
- 4-3. 배포 URL(Uniform Resource Locator)이 바로 열리지 않을 때
- 4-4. 모바일(mobile) 화면이 다르게 보일 때
- 4-5. Sigil로 출판 형태를 만드는 흐름
- 4-6. 다음 Codex 실습서로 넘어가기 전에

### 캡처(capture) 방향

- 로그인과 권한 화면은 개인정보 제거 기준을 적용합니다.
- 배포 대기와 배포 완료 화면을 구분합니다.
- 모바일(mobile) 화면 차이는 실제 모바일(mobile) 뷰포트 기준으로 확인합니다.
- Sigil 단계는 최종 출판 전환 흐름만 설명하고, 본문 실습 범위와 섞지 않습니다.

## 제작 원칙

- 결과 이미지는 실제 Playwright 캡처(capture)를 기준으로 합니다.
- 같은 이미지는 다시 사용하지 않습니다.
- ChatGPT 대화창 안 미리보기는 최종 결과 이미지로 쓰지 않습니다.
- HTML(HyperText Markup Language) 결과는 실제 브라우저(browser) 또는 배포 URL(Uniform Resource Locator)에서 확인합니다.
- HTML(HyperText Markup Language) 결과물이 만들어지면 Git에 푸시하고 배포 URL(Uniform Resource Locator)을 제공합니다.
- 독자가 직접 확인하기 어려운 리모트 환경을 전제로, 작업자는 배포 URL(Uniform Resource Locator) 확인까지 책임집니다.
