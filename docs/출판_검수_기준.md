# 바이브코딩(vibe coding) 1차 GPT(Generative Pre-trained Transformer) 입문서 출판 검수 기준

## 검수 목표

이 기준은 책이 독자에게 “책을 따라 하면 책에 실린 HTML(HyperText Markup Language) 캡처(capture)와 최대한 같은 타로(tarot) 사이트를 완성한다”는 경험을 안정적으로 전달하는지 확인하기 위한 기준입니다.

문장, 디자인, 프롬프트(prompt) 박스, 이미지, Playwright 캡처(capture), Git 푸시, 배포 URL(Uniform Resource Locator), Sigil EPUB(Electronic Publication) 전환 흐름을 함께 검수합니다.

## 절대 실패 기준

다음 항목 중 하나라도 있으면 출판 전 수정합니다.

- 깨진 이미지가 있습니다.
- 같은 이미지가 서로 다른 장면에서 반복됩니다.
- 본문 프롬프트(prompt) 박스와 같은 내용의 이미지가 반복됩니다.
- Chapter 1에 Chapter 2 또는 Chapter 3 결과 이미지가 다시 나옵니다.
- 카드나 결과 화면을 자체 SVG 대체 이미지로 보여줍니다.
- ChatGPT 대화창 안 미리보기 화면을 최종 결과물처럼 보여줍니다.
- Playwright 입력 캡처(capture) 없이 결과만 보여줍니다.
- HTML(HyperText Markup Language) 결과 캡처(capture) 없이 설명만으로 넘어갑니다.
- 결과물 데스크톱(desktop) 캡처(capture)는 있지만 모바일(mobile) 캡처(capture)가 없습니다.
- 독자에게 터미널, CLI, 로컬 편집기 사용을 요구합니다.
- 독자에게 직접 파일을 만들거나 확장자를 바꾸라고 안내합니다.
- 본문에 긴 코드 블록이 노출됩니다.
- 계정명, 이메일, 최근 대화, 저장소 목록 같은 개인정보가 보입니다.
- Git 푸시나 배포 URL(Uniform Resource Locator) 확인 없이 결과가 완성된 것처럼 설명합니다.
- 검증하지 않은 화면을 성공한 결과처럼 설명합니다.

## 디자인 검수 기준

- `디자인_가이드.md`의 화이트 기반 실습형 반응형 전자책 기준을 따릅니다.
- 이미지 중심 흐름을 유지하되, 중복 이미지는 제거합니다.
- Chapter 2와 Chapter 3의 장면 구조는 설명, 프롬프트(prompt) 박스, HTML(HyperText Markup Language) 결과 이미지 순서를 반복합니다.
- 프롬프트(prompt) 박스는 본문 설명과 분리되어야 합니다.
- 복사하기 버튼은 프롬프트(prompt) 박스 우측 상단에 있어야 합니다.
- 모바일(mobile)에서도 설명, 프롬프트(prompt) 박스, 이미지 순서가 깨지지 않아야 합니다.
- 데스크톱(desktop) 본문 폭이 지나치게 넓어지면 안 됩니다.
- Sigil EPUB(Electronic Publication) 전환 시 본문과 이미지 순서가 유지되어야 합니다.

## 이미지 검수 기준

- 모든 이미지 경로는 실제 파일로 연결되어야 합니다.
- 이미지 파일은 장면별로 고유 목적을 가져야 합니다.
- 같은 이미지 파일 경로는 책 전체에서 한 번만 사용해야 합니다.
- 이미지가 본문 설명과 맞지 않으면 교체합니다.
- 캡션은 독자가 이미지 목적을 바로 알 수 있게 씁니다.
- 캡처(capture) 안의 글자와 버튼은 모바일(mobile)에서도 읽혀야 합니다.
- 동그라미 번호는 UI(User Interface) 글자나 버튼을 가리지 않아야 합니다.
- 개인정보가 보이는 화면은 출판용 편집본에서 제거합니다.

## ChatGPT와 Playwright 캡처(capture) 기준

- Chapter 2와 Chapter 3은 Playwright가 ChatGPT에 실제 프롬프트(prompt)를 입력한 화면을 포함합니다.
- Playwright 입력 캡처(capture)와 HTML(HyperText Markup Language) 결과 캡처(capture)를 구분합니다.
- ChatGPT 응답 화면은 과정 설명으로만 사용합니다.
- 완성 결과는 ChatGPT 밖 실제 브라우저(browser) 또는 배포 URL(Uniform Resource Locator)에서 보여줍니다.
- 캡처(capture) 자동화가 막히면 수동 확인 컷으로 분류하고 사장님 검수 대상으로 올립니다.
- ChatGPT 최근 대화 목록과 계정 영역은 출판용 편집본에서 제거합니다.

## 프롬프트(prompt) 박스 검수 기준

- 본문에는 핵심 3줄만 보입니다.
- 전문은 복사하기 버튼으로 제공됩니다.
- 프롬프트(prompt)와 설명 문장이 섞이면 실패입니다.
- 프롬프트(prompt) 박스가 코드 블록처럼 길게 보이면 수정합니다.
- Chapter 2와 Chapter 3의 실습 단위에는 프롬프트(prompt) 박스가 있어야 합니다.

## HTML(HyperText Markup Language) 결과 재현 검수 기준

- Chapter 2와 Chapter 3에서 생성된 HTML(HyperText Markup Language) 결과는 책에 실리는 캡처(capture)와 기능, 화면 흐름, UI(User Interface), UX(User Experience)가 최대한 일치해야 합니다.
- LLM(Large Language Model) 특성상 문장이나 세부 결과가 달라질 수 있지만 핵심 기능과 화면 구조는 유지되어야 합니다.
- Chapter 3 타로(tarot)앱은 `home`, `ask`, `spread`, `result` 흐름을 가져야 합니다.
- 메이저 22장 카드 구조가 반영되어야 합니다.
- 5영역 리딩 결과가 반영되어야 합니다.
- 정방향과 역방향 결과가 반영되어야 합니다.
- 오늘 결과와 최근 이력 저장 흐름이 반영되어야 합니다.
- 최종 타로(tarot)앱은 데스크톱(desktop)과 모바일(mobile)에서 모두 확인합니다.

## Git 푸시와 배포 URL(Uniform Resource Locator) 검수 기준

- HTML(HyperText Markup Language) 결과물이 만들어지면 Git에 푸시하는 흐름이 있어야 합니다.
- 배포 URL(Uniform Resource Locator)을 제공해야 합니다.
- 사용자가 리모트 환경에서 직접 확인하기 어렵다는 전제를 반영합니다.
- 작업자는 배포 URL(Uniform Resource Locator)이 실제로 열리는지 확인해야 합니다.
- 데스크톱(desktop)과 모바일(mobile)에서 배포 URL(Uniform Resource Locator)을 확인합니다.
- Chapter 2의 Git, GitHub, ChatGPT 로그인 설명은 구글(Google) 계정 기준이어야 합니다.
- 계정명, 이메일, 저장소 목록 등 개인정보는 제거합니다.

## Sigil EPUB(Electronic Publication) 전환 검수 기준

- 디자인과 본문 구조가 Sigil EPUB(Electronic Publication) 전환을 방해하지 않아야 합니다.
- 복잡한 JavaScript 의존으로 본문 의미가 사라지면 안 됩니다.
- 프롬프트(prompt) 전문은 EPUB(Electronic Publication)에서도 보존되어야 합니다.
- 이미지 경로와 캡션은 EPUB(Electronic Publication) 전환 전 확인합니다.
- 모바일(mobile) 리더기에서 본문과 이미지 순서가 유지되어야 합니다.

## 문장 검수 기준

- 존댓말을 유지합니다.
- 독자 호칭은 “여러분” 또는 “독자분”으로 통일합니다.
- 과도하게 쉽다고 단정하거나 새로움을 과장하는 표현을 피합니다.
- 한 문장은 가능한 80자를 넘기지 않습니다.
- 한 단락은 한 생각만 담습니다.
- 영어 UI(User Interface) 이름은 처음 등장할 때 한국어 의미를 함께 설명합니다.

## 독자 이탈 위험 기준

다음 상황은 Reviewer가 집중 확인합니다.

- 첫 화면에서 무엇을 눌러야 할지 모릅니다.
- 영어 화면에서 어떤 단어를 봐야 할지 모릅니다.
- 권한 허용 화면에서 눌러도 되는지 불안합니다.
- 결과가 책과 다르게 나왔을 때 다음 행동을 모릅니다.
- 모바일(mobile) 화면에서 글자가 작거나 버튼이 잘립니다.
- 코드가 보여서 독자가 직접 해야 한다고 느낍니다.
- 이미지 없이 설명만 길게 이어집니다.

- Chapter 1에는 결과물 이미지가 없고, Chapter 2와 Chapter 3으로 넘어가는 연결만 있습니다.
- Chapter 2에서 독자가 이미지, 프롬프트(prompt), HTML(HyperText Markup Language) 결과, Git 연결, 배포 URL(Uniform Resource Locator) 흐름을 이해합니다.
- Chapter 3에서 독자가 타로(tarot)앱 제작 흐름을 화면 단위로 이해합니다.
- Chapter 4에서 막혔을 때 다시 확인할 위치를 찾을 수 있습니다.

- 디자인 가이드 기준이 검수 기준에 반영되어 있습니다.

- Playwright 입력 캡처(capture)와 HTML(HyperText Markup Language) 결과 캡처(capture) 기준이 모두 반영되어 있습니다.
