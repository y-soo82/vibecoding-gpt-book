# 바이브코딩(vibe coding) 1차 GPT(Generative Pre-trained Transformer) 입문서 디자인 가이드

## 디자인 목표

이 책의 디자인 목표는 독자가 화면을 보며 그대로 따라 할 수 있는 반응형 전자책을 만드는 것입니다.

디자인은 장식보다 이해를 우선합니다. 글보다 캡처(capture) 이미지가 많은 책이므로, 독자가 설명, 프롬프트(prompt), 결과 이미지를 같은 흐름으로 반복해서 읽을 수 있어야 합니다.

## 핵심 원칙

1. 깔끔하게 구성합니다.
2. 심플하게 반복합니다.
3. 모바일(mobile)에서도 읽기 쉽게 만듭니다.
4. 화면 크기에 따라 자연스럽게 배치합니다.

## 추천 디자인 콘셉트

이 책은 화이트 기반의 깔끔한 실습형 전자책으로 디자인합니다.

- 배경은 흰색 또는 아주 연한 회색을 사용합니다.
- 본문은 짙은 회색 계열을 사용합니다.
- 강조색은 한 가지 색만 사용합니다.
- 프롬프트(prompt) 박스, 본문, 이미지가 서로 분명히 구분되게 합니다.
- 장식, 그림자, 아이콘, 그라데이션은 최소화합니다.
- 독자가 따라 하는 순서가 화면에서 바로 보이게 합니다.

## 컬러 기준

| 용도 | 권장 색상 | 설명 |
|---|---|---|
| 기본 배경 | `#ffffff` | 본문 기본 배경입니다. |
| 보조 배경 | `#f7f7f8` | 프롬프트(prompt) 박스와 보조 영역에 사용합니다. |
| 본문 텍스트 | `#1f2933` | 긴 글을 읽기 위한 기본 색입니다. |
| 보조 텍스트 | `#667085` | 캡션과 보조 설명에 사용합니다. |
| 구분선 | `#e5e7eb` | 섹션 구분과 박스 선에 사용합니다. |
| 강조색 | `#2563eb` | 버튼, 링크, 현재 행동 안내에만 사용합니다. |

강조색은 한 화면에서 많이 쓰지 않습니다. 중요한 행동, 복사하기 버튼, 링크처럼 독자가 눌러야 하거나 확인해야 하는 위치에만 사용합니다.

## 글꼴 기준

- 본문은 시스템 기본 한글 글꼴을 사용합니다.
- 권장 순서: `-apple-system`, `Apple SD Gothic Neo`, `Pretendard`, `Noto Sans KR`, `sans-serif`
- 제목은 본문보다 굵고 선명하게 처리합니다.
- 장식적인 글꼴은 표지나 큰 챕터 제목에만 제한적으로 사용합니다.
- 모바일(mobile) 본문 크기는 16px 이상을 기준으로 합니다.
- 데스크톱(desktop) 본문 크기는 17px 전후를 기준으로 합니다.
- 줄간격은 1.6~1.7 범위로 둡니다.

## 본문 레이아웃

본문은 좁고 선명하게 읽히는 단일 흐름을 기준으로 합니다.

- 데스크톱(desktop) 본문 최대 폭은 880px 전후로 제한합니다.
- 모바일(mobile)은 좌우 여백 16px 전후를 기준으로 합니다.
- 한 문단에는 한 생각만 담습니다.
- 한 화면에는 가능한 한 한 가지 행동만 배치합니다.
- 긴 표는 모바일(mobile)에서 피합니다.
- 설명이 길어지면 목록이나 짧은 단락으로 나눕니다.

## Chapter 2, Chapter 3 한 장면 구조

Chapter 2와 Chapter 3의 실습 장면은 다음 구조를 반복합니다.

1. 설명
2. 프롬프트(prompt) 박스
3. ChatGPT 입력 캡처(capture)
4. HTML(HyperText Markup Language) 결과 이미지
5. 확인 문장

이 순서를 임의로 바꾸지 않습니다. 데스크톱(desktop)과 모바일(mobile)에서도 같은 순서로 읽히게 합니다.

## 프롬프트(prompt) 박스 기준

프롬프트(prompt) 박스는 독자가 ChatGPT에 입력할 내용을 분리해서 보여주는 영역입니다.

- 상단 왼쪽에는 “프롬프트(prompt)” 라벨을 둡니다.
- 상단 오른쪽에는 “복사하기” 버튼을 둡니다.
- 본문에는 핵심 3줄만 보이게 합니다.
- 전문은 복사하기 버튼으로 복사할 수 있게 합니다.
- 프롬프트(prompt) 박스는 연한 배경과 왼쪽 강조선으로 본문과 구분합니다.
- 프롬프트(prompt)와 설명 문장이 섞이지 않게 합니다.

## 복사하기 버튼 기준

- 버튼은 작지만 모바일(mobile)에서 누르기 쉬운 크기여야 합니다.
- 버튼 위치는 프롬프트(prompt) 박스 우측 상단으로 고정합니다.
- 버튼 문구는 “복사하기”로 통일합니다.
- 버튼이 프롬프트(prompt) 본문과 겹치면 안 됩니다.
- EPUB(Electronic Publication) 전환 시 버튼 기능이 제한될 수 있으므로, 전문 프롬프트(prompt)를 보존할 별도 구조를 함께 설계합니다.

## 이미지와 캡처(capture) 배치 기준

이미지는 이 책의 중심 요소입니다.

- 한 이미지는 한 목적만 가져야 합니다.
- 같은 이미지를 여러 장면에서 반복하지 않습니다.
- 캡처(capture) 아래에는 짧고 직접적인 캡션을 둡니다.
- 캡처(capture) 안의 글자가 모바일(mobile)에서도 읽힐 수 있어야 합니다.
- 버튼, 입력창, 주요 결과가 잘리면 안 됩니다.
- ChatGPT 입력 캡처(capture)와 HTML(HyperText Markup Language) 결과 캡처(capture)를 구분합니다.
- 결과 이미지는 실제 브라우저(browser) 또는 배포 URL(Uniform Resource Locator)에서 확인한 화면이어야 합니다.
- ChatGPT 대화창 안 미리보기는 최종 결과 이미지로 사용하지 않습니다.

## 모바일(mobile) 기준

모바일(mobile) 화면을 기본 독서 환경으로 봅니다.

- 설명, 프롬프트(prompt) 박스, 결과 이미지는 세로로 쌓습니다.
- 좌우 여백은 좁지만 답답하지 않게 유지합니다.
- 프롬프트(prompt) 박스의 복사하기 버튼은 손가락으로 누르기 쉬워야 합니다.
- 이미지가 화면 밖으로 넘치지 않아야 합니다.
- 가로 스크롤이 필요한 표는 피합니다.
- 캡처(capture) 글자가 작아지면 캡처(capture)를 분할합니다.

## 데스크톱(desktop) 기준

데스크톱(desktop)에서는 넓은 화면을 모두 쓰지 않습니다.

- 본문 최대 폭을 제한합니다.
- 실습 흐름은 모바일(mobile)과 같은 순서를 유지합니다.
- 2단 레이아웃은 원칙적으로 사용하지 않습니다.
- 이미지가 너무 크게 늘어나 흐려지지 않게 합니다.
- 본문과 이미지 사이 여백을 일정하게 유지합니다.

## Sigil EPUB(Electronic Publication) 전환 기준

최종 출판은 Sigil 기반 EPUB(Electronic Publication) 전환을 염두에 둡니다.

- CSS(Cascading Style Sheets)는 단순하게 유지합니다.
- 복잡한 JavaScript 의존을 줄입니다.
- 본문은 HTML(HyperText Markup Language) 문서만으로도 읽히는 구조여야 합니다.
- 프롬프트(prompt) 전문은 EPUB(Electronic Publication)에서도 사라지지 않아야 합니다.
- 이미지 경로, 캡션, 대체 텍스트를 EPUB(Electronic Publication) 전환 전 확인합니다.
- 모바일(mobile) 리더기에서 본문과 이미지 순서가 깨지지 않아야 합니다.

## 디자인 금지 사항

- 과한 장식
- 복잡한 카드 중첩
- 짙은 배경 위 긴 본문
- 작은 글씨 캡처(capture)
- 코드 블록 중심 구성
- 한 화면에 여러 행동을 동시에 배치
- 프롬프트(prompt) 전문을 본문에 길게 노출
- 모바일(mobile) 가로 스크롤이 생기는 표
- 이미지 없이 설명만 긴 페이지
- 캡처(capture)와 본문 설명이 서로 어긋나는 구성

## 디자인 검수 기준

- Chapter 2와 Chapter 3의 장면 구조가 반복되는지 확인합니다.
- 프롬프트(prompt) 박스가 설명과 분리되어 있는지 확인합니다.
- 복사하기 버튼이 모바일(mobile)에서 누르기 쉬운지 확인합니다.
- 캡처(capture) 이미지의 글자와 버튼이 읽히는지 확인합니다.
- 데스크톱(desktop)과 모바일(mobile)에서 읽는 순서가 같은지 확인합니다.
- EPUB(Electronic Publication) 전환 시 사라질 수 있는 정보가 없는지 확인합니다.
