---

## 1. 책의 한 줄 정의

이 책은 코딩을 가르치는 책이 아닙니다.

**웹서핑과 유튜브 시청 정도만 할 줄 아는 독자가 반응형 전자책을 보며 ChatGPT 대화창에 한 줄씩 부탁해서, 웹에서 실제로 열리는 사이트 2개를 완성하는 실습형 전자책**입니다.

독자가 책을 끝까지 따라 했을 때 손에 남아야 하는 결과는 지식이 아니라 주소입니다.

1. 이미지 한 장으로 만든 첫 웹사이트 주소
2. 타로카드 리딩 웹사이트 주소

### 책의 형태

이 책은 고정 PDF 한 가지로만 설계하지 않습니다.  
휴대전화·태블릿·컴퓨터에서 모두 읽을 수 있는 **반응형 웹 전자책**을 1차 형태로 둡니다.

독자가 휴대전화로 책을 열어 보면서 같은 휴대전화 또는 컴퓨터의 ChatGPT 화면을 따라갈 수 있어야 합니다. 따라서 본문, 이미지, 프롬프트 박스, 복사 버튼은 모바일 폭에서도 깨지지 않아야 합니다.

## 2. 핵심 판매 약속

판매 페이지와 책 첫머리에는 아래 약속을 반복해서 박습니다.

> 코딩을 몰라도 괜찮습니다.  
> 이 책에서는 ChatGPT 대화창에 한 줄씩 부탁하고, ChatGPT의 GitHub 연동으로 웹에 올립니다.  
> 끝까지 따라오시면 직접 만든 웹사이트 2개가 인터넷 주소로 남습니다.

### 독자가 사는 것

- HTML·CSS·JavaScript 지식이 아닙니다.
- 개발자 도구 사용법이 아닙니다.
- 터미널·명령줄·로컬 편집기 사용법이 아닙니다.
- **AI에게 부탁해서 결과물을 얻고, 그 결과를 웹에 공개하는 첫 경험**입니다.

### 유료 전자책으로서의 상품 구성

1. 본문 전자책
2. 첫 번째 사이트용 프롬프트
3. 타로 사이트용 화면별 프롬프트
4. 타로카드 예제 이미지 묶음
5. 완성 예시 사이트 주소
6. 막힐 때 쓰는 보강 프롬프트 모음

### 상품 형태

- 1차 상품: 반응형 웹 전자책
- 최종 패키징: Sigil을 통한 EPUB 전자책 제작
- 보조 상품: PDF 내보내기본
- 제공 자료: 프롬프트 원문, 타로 이미지 예제, 완성 사이트 주소
- 핵심 UX: 긴 프롬프트는 본문에서 3줄만 보이고, 우측 상단 **복사하기** 버튼으로 전문을 복사합니다.

## 3. 타겟 독자

### 1차 독자

- 웹서핑과 유튜브 시청은 할 수 있습니다.
- ChatGPT는 들어 보았지만, 무엇을 적어야 할지 자주 멈춥니다.
- GitHub는 처음 보거나 이름만 들어 보았습니다.
- 코딩, HTML, CSS, JavaScript, 저장소, 배포라는 단어가 낯섭니다.
- GitHub 화면을 직접 많이 만지는 것보다, ChatGPT에게 말로 부탁하는 흐름이 더 안전하다고 느낍니다.
- 컴퓨터보다 휴대전화 화면에 더 익숙할 수 있습니다.
- 영어 UI, 계정 가입, 권한 허용, 공개 설정에서 불안감을 느낍니다.

### 독자 심리

이 독자는 “코딩을 공부하고 싶다”보다 “나도 하나 만들어 보고 싶다”에 가깝습니다.  
따라서 설명보다 먼저 성취가 와야 합니다.

첫 10분 안에 독자는 아래 감각을 받아야 합니다.

> 아, 이건 제가 외우는 책이 아니라 보고 따라 하는 책이군요.

## 5. 시장 배경 장의 역할

AI와 바이브코딩 소개 장은 길게 쓰지 않습니다. 이 장의 목적은 독자를 설득하는 것이 아니라, “지금 이 책을 펼쳐도 늦지 않았다”는 감각을 주는 것입니다.

### 넣을 메시지

1. ChatGPT는 이미 대중 서비스가 되었습니다.
2. 개발자들은 AI 도구를 빠르게 쓰고 있습니다.
3. 그러나 일반인이 AI로 직접 웹사이트를 만드는 시장은 아직 초입입니다.
4. 그래서 지금은 코딩을 다 배우기보다, 먼저 AI에게 부탁해서 결과물을 만들어 보는 감각이 중요합니다.

### 통계 사용 원칙

통계는 출간 직전에 다시 확인합니다. 숫자는 본문 중심이 아니라 신뢰 보강용 박스로만 씁니다.

현재 참고 가능한 출처:

- OpenAI는 2025년 ChatGPT가 주간 활성 사용자 7억 명 규모에 도달했다고 밝혔고, 2026년 2월에는 9억 명 규모 보도도 나왔습니다. 본문에는 “수억 명이 매주 쓰는 대중 서비스” 정도로 표현합니다.  
  출처: https://openai.com/blog/how-people-are-using-chatgpt/ , https://techcrunch.com/2026/02/27/chatgpt-reaches-900m-weekly-active-users/
- Stack Overflow 2025 Developer Survey는 49,000명 이상 개발자를 대상으로 했고, AI 도구 사용·도입 의향이 매우 높지만 신뢰는 낮다는 흐름을 보여 줍니다. 본문에는 “개발자도 AI를 쓰지만, 검수와 판단은 여전히 중요하다”는 메시지로 씁니다.  
  출처: https://survey.stackoverflow.co/2025 , https://stackoverflow.blog/2026/02/18/closing-the-developer-ai-trust-gap/
- Microsoft 2025 Annual Report는 GitHub Copilot이 2천만 명 이상 사용자 규모에 도달했다고 설명합니다. 본문에는 “AI 코딩 도구는 이미 개발자 세계에서 빠르게 퍼지고 있다” 정도로만 씁니다.  
  출처: https://annualreports.ai/wp-content/uploads/annual-report-microsoft-2025.pdf

### 본문 표현 예시

> AI는 이미 많은 사람이 쓰는 도구가 되었습니다.  
> 개발자들도 AI에게 코드를 부탁하고, 결과를 검토하며 일합니다.  
> 다만 일반 독자가 ChatGPT 대화창만으로 직접 웹사이트를 만들어 보는 시장은 아직 초입입니다.  
> 그래서 지금 중요한 것은 코딩 지식 전체를 외우는 일이 아니라, AI에게 어떻게 부탁하면 결과가 나오는지 한 번 몸으로 겪어 보는 일입니다.

## 6. 이미지 60~70% 원칙

이 책의 핵심 형식은 **이미지 먼저, 글은 짧게**입니다.

### 페이지 비율

- 이미지: 60~70%
- 글: 30~40%
- 한 화면 설명: 캡처 1장 + 번호 + 짧은 설명
- 한 페이지 글자 수: 가능하면 300~500자
- 긴 설명은 본문이 아니라 표, 번호, 체크박스로 나눕니다.
- 전체 분량 목표: 80~100페이지
- 한 페이지는 모바일 화면 한두 번 스크롤 안에서 이해되도록 설계합니다.

### 이미지 종류

1. 전체 화면 캡처
2. 영역별 클로즈업 캡처
3. 결과 화면 캡처
4. 모바일 화면 캡처
5. 타로카드 샘플 이미지
6. 짧은 인포그래픽

### 이미지 사용 규칙

- 독자가 눌러야 하는 위치는 번호로 표시합니다.
- 번호는 캡처 위에 과하게 크게 박지 않습니다. `<input>` 같은 실제 UI 태그·버튼 글자보다 약 20% 작게 잡아, 가독성은 유지하되 화면을 덮지 않게 합니다.
- 번호는 버튼·입력칸·본문 글자를 가리지 않고, 대상 요소의 바로 옆에 둡니다.
- 같은 이미지를 반복 사용하지 않습니다.
- 반복되는 이미지나 반복되는 설명으로 콘텐츠 품질이 낮아지지 않게 합니다. 같은 화면을 다시 쓸 때는 반드시 목적이 달라야 합니다.
- 깨진 이미지가 나오면 출고 불가입니다.
- 결과물은 ChatGPT 대화창 내부 미리보기가 아니라, 실제 브라우저에서 열린 화면으로 보여 줍니다.

### 페이지 기본 템플릿

```md
## 지금 하실 일

![화면 캡처](images/...)

① 이 자리를 누릅니다.  
② 이 문장을 그대로 붙여 넣습니다.  
③ 기다린 뒤 결과 화면을 확인합니다.

> 막히면 이렇게 물어보세요:
> 방금 단계에서 어느 단추를 눌러야 하는지 한 줄씩 다시 알려 주세요.
```

## 7. 반응형 전자책 UX 원칙

### 7.1 기본 형태

이 책은 웹 페이지처럼 읽히는 반응형 전자책입니다.

- 모바일: 한 단 구성
- 태블릿: 본문 중심 + 여백 넓힘
- 데스크톱: 최대 폭 제한 + 좌우 여백
- 이미지: 화면 폭에 맞춰 줄어들되 글자가 읽히는 해상도 유지
- 표: 모바일에서 가로로 밀리지 않게 카드형 목록으로 전환

### 7.2 프롬프트 박스 UX

책에 실리는 프롬프트는 길 수 있습니다. 긴 프롬프트를 본문에 전부 펼치면 이미지 중심 책의 리듬이 무너집니다.

따라서 모든 긴 프롬프트는 아래 규칙을 따릅니다.

1. 본문에는 앞 3줄 정도만 보입니다.
2. 박스 우측 상단에 **복사하기** 버튼을 둡니다.
3. 독자가 버튼을 누르면 전문이 클립보드에 복사됩니다.
4. 복사 후 버튼 문구는 잠깐 **복사됨** 으로 바뀝니다.
5. 전문을 보고 싶은 독자를 위해 **전체 보기** 토글을 둘 수 있습니다.

### 7.3 프롬프트 박스 예시

```html
<div class="prompt-card">
  <button class="copy-button">복사하기</button>
  <div class="prompt-preview">
    이 이미지와 어울리는 반응형 웹사이트를 만들어 주세요.
    휴대전화와 컴퓨터에서 보기 좋게 해 주세요.
    따뜻한 분위기의 한 페이지로 구성해 주세요.
  </div>
  <textarea class="prompt-full" hidden>전문 프롬프트...</textarea>
</div>
```

본문에는 이 HTML을 그대로 노출하지 않습니다. 제작 지침으로만 사용합니다.

### 7.4 복사 버튼 필수 검수

- 모바일에서 버튼이 손가락으로 누르기 쉬워야 합니다.
- 복사 버튼이 프롬프트 첫 줄을 가리지 않아야 합니다.
- 복사 실패 시 “직접 길게 눌러 복사해 주세요” 안내가 떠야 합니다.
- PDF 내보내기본에서는 복사 버튼이 작동하지 않을 수 있으므로, QR 또는 별도 프롬프트 페이지 링크를 제공합니다.

## 8. 분량 설계 — 80~100페이지

전체 목표는 80~100페이지입니다.  
이미지 중심 전자책이므로 페이지 수는 글자 수보다 **화면 단계 수**로 계산합니다.

### 권장 페이지 배분

| 구간 | 목표 페이지 |
|---|---:|
| 머리말·책 사용법 | 6~8 |
| Chapter 0. AI와 바이브코딩 시장 소개 | 8~10 |
| Chapter 1. ChatGPT 대화창 알아보기 | 10~12 |
| Chapter 2. 첫 번째 사이트 만들기 | 10~12 |
| Chapter 3. 첫 번째 사이트 배포 | 10~12 |
| Chapter 4. 타로 사이트 미리보기·준비 | 8~10 |
| Chapter 5. 타로 이미지 준비 | 8~10 |
| Chapter 6. 타로 사이트 만들기 | 18~24 |
| Chapter 7. 타로 사이트 배포 | 8~10 |
| 부록 | 6~8 |

예상 합계: 92~116페이지입니다.  
실제 편집에서는 Chapter 6을 압축하거나 부록을 줄여 80~100페이지 안에 맞춥니다.

### 분량이 부족할 때의 보강 우선순위

분량이 부족해도 무작정 새 주제를 늘리지 않습니다. 아래 순서로 보강합니다.

1. 타로 사이트 완성도를 높입니다.
   - 결과 화면 해석 품질
   - 모바일 화면 완성도
   - 카드 상세 팝업
   - 최근 본 카드 이력
   - 공유 문구
2. ChatGPT와 GitHub 연동·배포 대화 단계를 촘촘히 만듭니다.
3. ChatGPT 답변이 달라질 때 쓰는 보강 프롬프트를 늘립니다.
4. 그래도 부족하면 **세 번째 미니 사이트**를 부록 실습으로 추가합니다.

### 세 번째 미니 사이트 후보

세 번째 실습은 본문 핵심이 아니라 분량 보강용 후보입니다. 추가한다면 8~12페이지 안에서 끝나야 합니다.

권장 후보:

1. 가족 모임 안내장 사이트
   - 사진 1장 + 날짜·장소·연락처
   - 독자층에 가장 친숙합니다.
2. 나를 소개하는 한 페이지
   - 자기소개·사진·연락 링크
   - 첫 번째 사이트와 겹칠 수 있어 우선순위는 낮습니다.
3. 작은 메뉴판 사이트
   - 음식 사진 몇 장 + 가격표
   - 자영업 독자에게 좋지만 자료 준비가 필요합니다.

권장 판단: 우선은 **타로 사이트 완성도 강화**로 80~100페이지를 채우고, 세 번째 사이트는 부록 후보로만 둡니다.

## 9. 책 전체 목차 권장안

현재 초안을 참고하되, 유료 전자책 기준으로 아래 구조를 권장합니다.

아래 목차는 사용자가 준 큰 순서를 유지하되, 독자 입장에서 더 따라 하기 쉬운 순서로 다시 배열한 안입니다. 핵심은 “설명 → 메뉴 → 실습”이 아니라 **먼저 왜 하는지 알고, 필요한 화면만 익힌 뒤, 작은 성공을 얻고, 그다음 큰 실습으로 넘어가는 흐름**입니다.

### 머리말

- 이 책은 코딩 책이 아닙니다
- 이 책은 반응형 전자책입니다
- 이 책에서 쓰는 도구는 ChatGPT 대화창과 ChatGPT의 GitHub 연동입니다
- 책을 끝내면 웹에서 열리는 사이트 2개가 남습니다
- 이미지 60~70%로 보고 따라가는 책입니다
- 긴 프롬프트는 3줄만 보이고 복사하기 버튼으로 전문을 복사합니다

### Chapter 0. AI와 바이브코딩, 지금 어디까지 왔나요

목적: 독자가 “왜 지금 이 책인가”를 이해합니다.

- AI에게 말로 부탁해서 결과물을 받는 시대
- 바이브코딩이란 무엇인가요
- 개발자들은 이미 AI를 쓰고 있습니다
- 하지만 일반인이 직접 웹사이트를 만드는 시장은 아직 초입입니다
- 이 책에서는 코딩 공부보다 먼저 완성 경험을 합니다

이미지:

- AI 사용 흐름 인포그래픽
- “말로 부탁 → 결과 확인 → 다시 부탁 → GitHub 연동 배포” 흐름도
- 통계 요약 카드 1장

### Chapter 1. ChatGPT 대화창 알아보기

목적: 독자가 ChatGPT 화면에서 겁먹지 않습니다.

- 첫 화면 전체 보기
- 새 채팅
- 입력창
- 첨부 단추
- 보내기 단추
- 답변이 길게 나올 때 보는 자리
- 처음에는 몰라도 되는 메뉴들

주의:

메뉴 상세 소개를 너무 앞에서 길게 하지 않습니다.  
처음에는 실습에 필요한 자리만 보여 줍니다.

이미지:

- ChatGPT 전체 화면 1장
- 입력창 클로즈업 1장
- 첨부 메뉴 클로즈업 1장
- 답변 화면 1장

### 독자 관점 재배치 원칙

기존 초안의 목차는 주제는 맞지만, 독자가 중간에 지칠 수 있는 순서가 섞일 수 있습니다. 따라서 최종 편집에서는 아래 원칙을 적용합니다.

1. 시장 설명은 짧고 시각적으로 처리합니다.
2. ChatGPT 메뉴 상세 소개는 첫 실습 전에 모두 넣지 않습니다.
3. 첫 사이트를 빨리 만들어 “성공 감각”을 먼저 줍니다.
4. GitHub 연동 배포는 이미지가 많은 별도 장으로 둡니다.
5. 타로 실습은 완성 결과 미리보기 후 진행합니다.
6. 긴 프롬프트는 읽는 대상이 아니라 복사하는 대상으로 취급합니다.
7. 독자가 외워야 하는 내용보다 눌러야 하는 위치를 우선합니다.

### Chapter 2. 첫 번째 사이트 만들기 — 이미지 한 장으로 웹사이트 받기

목적: 첫 성취를 빠르게 줍니다.

- 좋아하는 이미지 한 장 고르기
- ChatGPT에 이미지 첨부하기
- 첫 프롬프트 보내기
- 결과 받기
- 마음에 안 드는 부분을 한 줄로 고치기
- 컴퓨터와 휴대전화에서 결과 화면 비교하기

핵심 프롬프트:

> 이 이미지와 어울리는 반응형 웹사이트를 만들어 주세요.  
> 휴대전화와 컴퓨터에서 보기 좋게 해 주세요.

이미지:

- 첨부할 이미지 예시
- 이미지 첨부 직전 화면
- ChatGPT 답변 화면
- 완성 사이트 데스크톱 화면
- 완성 사이트 모바일 화면

### Chapter 3. 첫 번째 사이트를 웹에 올리기

목적: 독자가 “내 사이트 주소”를 얻습니다.

- 왜 GitHub를 쓰나요
- GitHub 가입
- GitHub 로그인
- ChatGPT에서 GitHub 연동하기
- ChatGPT에게 새 저장소 생성과 업로드를 부탁하기
- ChatGPT에게 GitHub Pages 활성화를 부탁하기
- 최종 주소 확인
- 휴대전화에서 열어 보기

주의:

독자가 GitHub 화면에서 개발자처럼 저장소·파일·Pages 설정을 모두 직접 조작하게 하지 않습니다.  
기본 흐름은 **ChatGPT에 GitHub를 연동한 뒤, 대화창에서 “새 저장소를 만들고 Pages까지 켜 주세요”라고 부탁하는 방식**입니다.

ChatGPT 계정·요금제·권한 상태에 따라 직접 배포가 막히면, 그때만 GitHub 화면 단계 안내로 우회합니다. 이 우회 흐름은 본문 마지막에 “연동이 안 될 때” 보조 경로로 둡니다.

이미지:

- GitHub 메인 화면
- 가입 화면
- 로그인 화면
- ChatGPT 설정·커넥터 화면
- GitHub 연결 승인 화면
- ChatGPT에 배포를 부탁하는 대화 화면
- ChatGPT가 최종 주소를 알려 주는 화면
- 필요 시 보조 경로: Pages 설정 화면
- 휴대전화 결과 화면

### Chapter 4. 두 번째 실습 준비 — 타로 사이트는 어떻게 생겼나요

목적: 독자가 만들 결과를 먼저 봅니다.

- 타로가 무엇인지 짧게 설명
- 우리가 만들 사이트 화면 흐름
- 주제 선택
- 질문 입력
- 카드 뽑기
- 결과 보기
- 공유하기

이미지:

- 완성 타로 사이트 전체 흐름 1장
- 화면 흐름도 1장
- 결과 화면 미리보기 1장

### Chapter 5. 타로카드 이미지는 어떻게 준비하나요

목적: 이미지 생성 한도 때문에 독자가 중간에 막히지 않게 합니다.

- 타로카드는 78장입니다
- ChatGPT 이미지 생성은 무료 한도가 있습니다
- 그래서 책에서는 2장만 샘플로 만들어 봅니다
- 나머지 카드는 제공 자료를 사용합니다
- 제공 자료를 쓰는 이유는 실습을 끝까지 진행하기 위해서입니다

샘플 이미지 생성:

1. 마법사 카드
2. 태양 카드

주의:

기존 초안은 샘플 3장으로 되어 있으나, 유료 독자의 이미지 한도와 실패 가능성을 고려해 **샘플 2장**으로 줄입니다. 연인 카드는 결과물 예시 또는 제공 자료에만 둡니다.

이미지:

- 샘플 카드 2장
- 제공 카드 묶음 미리보기
- “직접 만들기 vs 제공 자료 사용” 비교표

본문 문장 예시:

> 이 책에서는 78장을 모두 직접 만들지 않습니다.  
> 이미지 생성에는 한도가 있고, 78장을 모두 만들다 보면 중간에 멈출 수 있습니다.  
> 그래서 두 장만 직접 만들어 감각을 익히고, 나머지는 책과 함께 제공되는 카드 이미지를 사용합니다.

이미지:

## 10. 이미지 영역 사전 기획

이미지 중심 전자책에서는 본문을 먼저 쓰고 나중에 이미지를 끼우면 안 됩니다.  
각 장은 먼저 **필수 이미지 컷 리스트**를 확정한 뒤, 그 이미지 사이를 짧은 문장으로 연결합니다.

### 10.1 이미지 컷의 종류

| 종류 | 쓰임 |
|---|---|
| 전체 화면 | 독자가 지금 어느 화면에 있는지 확인 |
| 영역 클로즈업 | 눌러야 할 버튼·입력칸·설정 위치 안내 |
| 결과 화면 | 제대로 따라 했을 때 보이는 모습 확인 |
| 모바일 확인 | 휴대전화에서 실제로 열리는지 확인 |
| 흐름도 | 여러 단계를 한눈에 보여 주는 요약 |
| 문제 상황 | 막혔을 때 보이는 화면과 해결 안내 |

### 10.2 이미지 배치 규칙

- 한 행동에는 최소 1장 이상의 이미지가 있어야 합니다.
- 독자가 처음 보는 화면은 반드시 전체 화면 1장을 먼저 둡니다.
- 그다음 눌러야 하는 곳은 클로즈업으로 다시 보여 줍니다.
- “연동”, “권한 허용”, “배포”, “주소 확인”처럼 불안한 단계는 이미지 없이 글만으로 설명하지 않습니다.
- 긴 설명은 이미지 아래 캡션 1~2줄로 제한합니다.
- 번호가 들어간 이미지는 본문 번호와 반드시 일치해야 합니다.
- 같은 캡처를 다시 쓰지 않습니다. 꼭 같은 화면을 다시 보여 줘야 하면 확대 위치, 표시 번호, 설명 목적을 다르게 합니다.
- 비슷한 설명이 2번 이상 반복되면 표나 체크리스트로 합쳐 콘텐츠 밀도를 높입니다.

### 10.3 Chapter 3 이미지 기획 — ChatGPT GitHub 연동으로 첫 사이트 배포

Chapter 3은 독자가 가장 불안해할 장입니다.  
GitHub라는 낯선 서비스와 “연동 권한” 화면이 나오기 때문입니다. 따라서 이 장은 거의 모든 단계에 이미지를 넣습니다.

목표 이미지 수: 14~18장

| 순서 | 이미지 컷 | 목적 | 필수 여부 |
|---:|---|---|---|
| 1 | GitHub 메인 화면 전체 | GitHub에 처음 들어온 자리 확인 | 필수 |
| 2 | GitHub 가입/로그인 버튼 클로즈업 | 어디로 들어갈지 안내 | 필수 |
| 3 | GitHub 가입 화면 | 이메일·비밀번호·사용자 이름 입력 자리 안내 | 필수 |
| 4 | 이메일 인증 화면 | 인증 코드 입력 불안 완화 | 권장 |
| 5 | GitHub 로그인 화면 | 다음 접속 때 쓰는 자리 안내 | 필수 |
| 6 | ChatGPT 첫 화면 전체 | 다시 ChatGPT로 돌아오는 흐름 확인 | 필수 |
| 7 | ChatGPT 설정 또는 커넥터 진입 화면 | GitHub 연동 위치 안내 | 필수 |
| 8 | 커넥터 목록에서 GitHub가 보이는 화면 | GitHub 연결 시작점 안내 | 필수 |
| 9 | GitHub 권한 승인 화면 | Authorize/승인 버튼 안내, 불안 완화 | 필수 |
| 10 | ChatGPT에 GitHub 연결 완료가 보이는 화면 | 연동 성공 확인 | 필수 |
| 11 | 첫 사이트 배포 요청 프롬프트 박스 | 독자가 복사할 문장 안내 | 필수 |
| 12 | ChatGPT가 저장소 생성·업로드를 진행하는 답변 화면 | “기다리는 자리” 안내 | 권장 |
| 13 | ChatGPT가 최종 주소를 알려 주는 화면 | 가장 중요한 성취 확인 | 필수 |
| 14 | 최종 주소를 브라우저에서 연 데스크톱 화면 | 실제 웹 공개 확인 | 필수 |
| 15 | 같은 주소를 휴대전화에서 연 화면 | 모바일 확인 | 필수 |
| 16 | ChatGPT가 직접 배포할 수 없다고 답한 예시 화면 | 보조 경로 진입 안내 | 권장 |
| 17 | 보조 경로용 GitHub Pages 설정 화면 | 연동 실패 시 대체 안내 | 선택 |
| 18 | Chapter 3 마무리 체크 이미지 | “주소가 열리면 성공” 요약 | 권장 |

### 10.4 Chapter 3 페이지 배치 예시

| 페이지 | 이미지 | 글 역할 |
|---:|---|---|
| 1 | GitHub 메인 전체 | GitHub는 사이트를 올려 두는 공개 책장이라고 설명 |
| 2 | 가입/로그인 클로즈업 | 처음이면 가입, 이미 있으면 로그인 |
| 3 | 가입 화면 | 입력칸별로 한 줄 안내 |
| 4 | 인증 화면 | 이메일 인증은 정상 절차라고 안심 |
| 5 | ChatGPT 설정 전체 | 다시 ChatGPT로 돌아와 연동 시작 |
| 6 | 커넥터 GitHub 클로즈업 | GitHub 연결 버튼 찾기 |
| 7 | 권한 승인 화면 | 승인해도 결제가 아니며, 배포에 필요한 권한이라고 설명 |
| 8 | 연결 완료 화면 | 이제 대화로 부탁할 준비 완료 |
| 9 | 배포 요청 프롬프트 3줄 미리보기 | 복사하기 버튼으로 전문 복사 |
| 10 | ChatGPT 진행 답변 | 기다리는 동안 새로고침하지 않기 |

### 10.5 Chapter 6 이미지 기획 — 타로 사이트 만들기

Chapter 6은 책의 가장 큰 실습입니다. 화면별로 동일한 이미지 구조를 반복해야 독자가 안정감을 느낍니다.

각 화면마다 아래 4장을 기본 단위로 둡니다.

1. 만들 결과 미리보기
2. ChatGPT에 보낼 프롬프트 박스
3. ChatGPT 답변 완료 화면
4. 실제 브라우저 결과 화면

타로 사이트 화면은 5개 이상이므로 기본 20장 안팎이 필요합니다.

필수 컷:

- 메인 화면 데스크톱
- 메인 화면 모바일
- 주제 선택 화면
- 질문 입력 화면
- 카드 뽑기 화면
- 카드 3장 선택 완료 화면
- 결과 화면
- 카드 상세 팝업
- 최근 이력 또는 다시 보기 화면
- 타로 사이트 최종 모바일 화면

### 10.6 이미지 파일명 계획

### 10.7 이미지 출고 기준

- 캡처 안 글자가 모바일에서도 읽혀야 합니다.
- 번호가 버튼이나 글자를 가리면 재촬영 또는 재합성합니다.
- 번호 크기는 대상 UI 글자보다 약 20% 작게 유지합니다. 너무 큰 번호로 화면을 덮으면 재합성합니다.
- 권한 승인 화면은 민감하므로 개인 계정 정보가 보이면 반드시 가립니다.
- 결과 화면은 실제 주소에서 열린 화면이어야 합니다.
- ChatGPT 답변 화면은 조작·합성하지 않습니다.
- 깨진 이미지, 빈 placeholder, TODO 문구가 보이면 출고 불가입니다.
- 같은 장 안에서 같은 이미지가 반복되면 출고 전 이유를 확인합니다. 이유가 없으면 제거하거나 새 캡처로 교체합니다.

### 10.8 원격 배포 확인과 캐시 무효화

작업자는 원격 환경에서 확인합니다. 따라서 로컬 미리보기만으로 완료 판단하지 않습니다.  
최종 확인은 반드시 GitHub Pages 또는 외부 공개 URL 배포 후 외부 접속으로 진행합니다.

캐시·쿠키 때문에 새로고침해도 수정 내역이 보이지 않는 일이 생기지 않도록 아래 규칙을 둡니다.

1. 배포용 `index.html`에는 캐시 방지 메타 태그를 넣습니다.
2. CSS·JS·이미지에는 버전 쿼리를 붙입니다. 예: `style.css?v=20260515-01`
3. Docsify 또는 정적 웹북 설정에는 `noCache`를 켭니다.
4. 배포 후 확인 URL에는 확인용 쿼리를 붙입니다. 예: `?v=20260515-01`
5. 외부 접속 확인은 일반 창 1회, 시크릿 창 1회, 모바일 화면 1회를 기준으로 합니다.
6. 수정 확인이 안 되면 먼저 버전 쿼리 변경 여부를 확인합니다.
7. 쿠키나 로그인 상태가 영향을 주는 화면은 캡처용 계정과 확인용 브라우저를 분리합니다.

출고 판단은 “내 컴퓨터에서 보인다”가 아니라 **외부 공개 URL에서 최신 내용이 보인다**를 기준으로 합니다.

## 12. 제작 원칙

### 12.1 이미지 우선

모든 장은 이미지가 먼저 와야 합니다.  
글은 이미지를 따라가는 보조 설명입니다.

나쁜 순서:

1. 긴 설명
2. 프롬프트
3. 결과 이미지

좋은 순서:

1. 만들 결과 이미지
2. 지금 누를 화면 이미지
3. 복사할 프롬프트
4. 결과 확인 이미지

### 12.2 행동 단위

한 페이지에는 가능한 한 하나의 행동만 둡니다.

- 가입하기
- 이미지 올리기
- 프롬프트 보내기
- 결과 보기
- ChatGPT에게 GitHub 배포 부탁하기
- 최종 주소 확인하기

### 12.3 불안 완화 문장

민감한 화면에는 반드시 안심 문장을 둡니다.

예시:

- 이 단추를 눌러도 결제가 진행되지는 않습니다.
- 이 이름은 나중에 다시 만들 수 있습니다.
- 지금은 공개 테스트용 저장소이므로 개인 정보를 넣지 않습니다.
- GitHub 연결 권한 화면에서는 저장소 생성·수정 권한이 필요할 수 있습니다. 개인 정보나 비밀 자료는 올리지 않습니다.
- 화면이 조금 달라도 오른쪽 위의 같은 단어를 찾으시면 됩니다.

### 12.4 프롬프트 중심

독자는 코드를 고치지 않습니다.  
독자가 하는 일은 ChatGPT에게 다시 부탁하는 것입니다.

예시:

> 글자가 너무 작습니다. 휴대전화에서 보기 좋게 조금 키워 주세요.  
> 버튼이 아래로 밀렸습니다. 한 화면 안에 들어오게 다시 정리해 주세요.  
> 카드 이미지가 깨져 보입니다. 이미지 경로를 다시 확인해서 고쳐 주세요.

### 12.5 프롬프트 표시 규칙

긴 프롬프트를 본문에 모두 펼치지 않습니다.

- 1~3줄: 본문에 보이는 미리보기
- 전문: 복사하기 버튼으로 복사
- 보조: 전체 보기 토글
- 모바일: 버튼은 우측 상단, 너무 작지 않게
- PDF: 프롬프트 전문 페이지 또는 QR 링크 제공

### 12.6 Sigil EPUB 제작 원칙

최종 전자책은 Sigil을 통해 EPUB로 제작합니다.

- 반응형 웹북 구조를 먼저 안정화합니다.
- 이후 Sigil에서 열 수 있는 EPUB 구조로 변환합니다.
- JavaScript 복사 버튼은 EPUB 리더에서 제한될 수 있으므로, EPUB 안에서는 프롬프트 전문 페이지 또는 링크를 함께 제공합니다.
- Sigil에서 목차, 이미지 경로, CSS, 표지, 메타데이터를 최종 검수합니다.
- EPUB 내 이미지 용량은 너무 크지 않게 압축하되, 캡처 글자가 흐려지면 안 됩니다.
- EPUB에서 복사 버튼이 작동하지 않는 리더를 대비해 각 프롬프트마다 “전문 보기” 대체 경로를 둡니다.

### 12.7 원격 확인 원칙

원격 환경에서는 최종 결과를 로컬 파일로 확인하지 않습니다.

- GitHub Pages 또는 공개 URL에 배포합니다.
- 외부 URL로 접속해 최신 내용이 보이는지 확인합니다.
- 캐시 무효화를 위해 버전 쿼리와 noCache 설정을 사용합니다.
- 수정 내역 확인용 빌드 번호를 화면 하단 또는 HTML 주석에 남깁니다.
- 같은 URL에서 이전 화면이 계속 보이면 캐시 문제로 판단하고, 쿼리 버전과 배포 시간을 먼저 확인합니다.

## 14. 기존 초안에서 바꿀 것

### 14.1 메뉴 설명 위치 조정

ChatGPT 메뉴 상세 소개를 너무 앞에 크게 두지 않습니다.  
첫 번째 실습 전에 필요한 것은 네 자리뿐입니다.

- 새 채팅
- 입력창
- 첨부
- 보내기

상세 메뉴는 첫 실습 뒤 또는 부록으로 보냅니다.

### 14.2 타로 이미지 생성 축소

기존 초안의 샘플 3장은 2장으로 줄입니다.  
무료 한도와 실패 가능성을 독자에게 먼저 설명합니다.

### 14.3 개발자 표현 제거

본문에서 아래 표현은 그대로 두지 않습니다.

- 파일명 노출
- 확장자 설명
- 코드 조각
- 100dvh 같은 CSS 단위
- images/cards 같은 경로

프롬프트 안에 꼭 필요하면 “그대로 복사하시면 됩니다. 뜻은 몰라도 괜찮습니다”라는 안내를 붙입니다.

### 14.4 결과 화면 품질 강화

타로 결과 화면은 책의 신뢰를 결정합니다.  
이미지가 깨지거나 카드가 작게 보이면 출고하지 않습니다.

필수 검수:

- 카드 이미지 깨짐 없음
- 모바일에서 글자 겹침 없음
- 결과 박스 잘림 없음
- 버튼이 눌릴 것처럼 보임
- 실제 사이트 주소에서 열림

### 14.5 프롬프트 전문 노출 방식 변경

현재 초안처럼 긴 프롬프트를 본문에 모두 펼치면 이미지 중심 전자책이 무거워집니다.  
따라서 화면별 프롬프트는 **3줄 미리보기 + 복사하기 버튼 + 전체 보기 토글** 구조로 바꿉니다.

프롬프트 원문은 별도 데이터 또는 숨김 영역에 보관합니다. 독자는 전문을 읽기보다 복사해서 ChatGPT에 붙여 넣는 흐름으로 갑니다.

### 14.6 반응형 전자책 검수 추가

기존 초안은 웹에서 볼 수 있으나, 출판 상품으로는 반응형 전자책 검수가 더 필요합니다.

필수 뷰포트:

- 모바일 세로 390px
- 모바일 큰 화면 430px
- 태블릿 768px
- 데스크톱 1280px

각 뷰포트에서 이미지, 표, 프롬프트 박스, 복사 버튼, 목차가 깨지지 않아야 합니다.

### 14.7 목차 재기획 기준

사용자가 준 목차는 큰 뼈대로 유지합니다.  
다만 최종 목차는 독자가 실제로 따라가기 좋은 순서로 한 번 더 재기획합니다.

재기획 기준:

- 첫 성취가 너무 늦게 나오지 않아야 합니다.
- 메뉴 설명이 실습보다 앞에서 과하게 길면 줄입니다.
- GitHub 연동 배포는 독자가 불안해하므로 독립 장으로 둡니다.
- 타로 실습은 완성 결과를 먼저 보여 준 뒤 프롬프트로 들어갑니다.
- 이미지 생성 한도 설명은 타로 이미지 만들기 전에 먼저 둡니다.
- 부록은 실패 상황 중심으로 구성합니다.

## 15. 판매 페이지 문구 초안

### 제목

보면서 따라 하는 바이브코딩 입문

### 부제

ChatGPT 대화창만으로 웹사이트 2개 만들기

### 한 줄 소개

웹서핑만 할 줄 알아도 괜찮습니다. 반응형 전자책을 보며 ChatGPT에게 한 줄씩 부탁해서 첫 웹사이트와 타로 사이트를 만들고, ChatGPT의 GitHub 연동으로 인터넷에 올려 봅니다.

### 상세 소개

이 책은 코딩 문법을 외우는 책이 아닙니다.  
ChatGPT 대화창에 무엇을 어떻게 부탁하면 되는지, 화면을 보면서 한 단계씩 따라 하는 책입니다.

첫 번째 실습에서는 이미지 한 장으로 작은 웹사이트를 만듭니다.  
두 번째 실습에서는 타로카드 리딩 사이트를 만듭니다.  
두 사이트 모두 ChatGPT에게 GitHub 배포를 부탁해 실제 인터넷 주소로 열어 봅니다.

책의 60~70%는 화면 이미지입니다.  
어디를 눌러야 하는지, 어떤 문장을 붙여 넣어야 하는지, 결과가 어떻게 보여야 하는지 눈으로 확인하며 따라가실 수 있습니다.

긴 프롬프트는 본문에 3줄만 보입니다.  
우측 상단의 복사하기 버튼을 누르면 전문이 복사되므로, 긴 문장을 직접 드래그하지 않으셔도 됩니다.

### 이미지

- [ ] 전체 책 기준 이미지 비율이 60~70%입니다.
- [ ] 깨진 이미지가 없습니다.
- [ ] 같은 이미지가 불필요하게 반복되지 않습니다.
- [ ] 반복 이미지·반복 설명 때문에 콘텐츠 품질이 떨어지는 구간이 없습니다.
- [ ] 번호 크기는 대상 UI 글자보다 약 20% 작고, 화면을 덮지 않습니다.
- [ ] Chapter 3 GitHub 연동 배포 장에 필수 이미지 컷이 모두 들어 있습니다.
- [ ] 모바일 화면 캡처가 포함되어 있습니다.
- [ ] 타로 결과 화면 카드가 모두 정상 표시됩니다.

### 반응형 전자책

- [ ] 모바일 390px에서 본문이 가로로 밀리지 않습니다.
- [ ] 태블릿 768px에서 이미지와 표가 자연스럽게 보입니다.
- [ ] 데스크톱 1280px에서 본문 폭이 과하게 넓어지지 않습니다.
- [ ] 긴 프롬프트는 3줄 미리보기로 보입니다.
- [ ] 복사하기 버튼이 모든 프롬프트 박스에서 작동합니다.
- [ ] 복사 후 “복사됨” 피드백이 보입니다.
- [ ] PDF 내보내기본에서도 프롬프트 전문 접근 방법이 있습니다.

### 실습

## 17. 권장 작업 순서

1. 현재 원고를 위 목차로 재배치합니다.
2. 독자 관점에서 목차를 한 번 더 재기획합니다.
3. Chapter 0 시장 배경 장을 짧게 새로 씁니다.
4. ChatGPT 메뉴 설명을 실습 전 필수 영역과 부록 영역으로 나눕니다.
5. 첫 사이트 실습을 “빠른 성취” 중심으로 압축합니다.
6. 배포 장을 ChatGPT GitHub 연동 대화 중심으로 다시 정리합니다.
7. Chapter 3 이미지 컷 리스트 14~18장을 먼저 촬영 계획으로 확정합니다.
8. 타로 샘플 이미지를 2장으로 줄이고, 제공 자료 사용 이유를 명확히 씁니다.
9. 타로 사이트 화면별 프롬프트와 결과 캡처를 다시 검수합니다.
10. 긴 프롬프트 박스를 3줄 미리보기 + 복사하기 버튼 구조로 바꿉니다.
11. 반복 이미지·반복 설명을 제거해 콘텐츠 밀도를 조정합니다.
12. 전체 페이지 수가 80~100페이지에 들어오는지 재계산합니다.
13. 분량이 부족하면 타로 사이트 완성도 강화부터 보강합니다.
14. Sigil EPUB 제작 경로를 별도로 점검합니다.
15. 배포 확인용 캐시 무효화 규칙을 적용합니다.
16. 부록에 보강 프롬프트를 모읍니다.
17. Reviewer agent로 50대 전후 입문 독자 검수를 돌립니다.
18. Verifier agent로 이미지 깨짐·링크·GitHub 연동 배포 흐름·복사 버튼·반응형 뷰포트·캐시 무효화를 검수합니다.

## 18. 최종 기획 판단

이 책은 “AI 시대의 코딩 입문서”보다 **“반응형 전자책을 보며 AI에게 부탁해서 내 결과물을 웹에 올려 보는 첫 경험 패키지”**로 포지셔닝해야 합니다.

유료 전자책으로서 가장 강한 문장은 이것입니다.

> 코딩을 배우기 전에, 먼저 하나 만들어 보세요.  
> ChatGPT 대화창만으로 웹사이트 2개를 만들고 인터넷에 올립니다.

이 방향이면 현재 초안의 장점인 실습성, 캡처 중심 구성, 타로 사이트 예제를 살리면서도 독자에게 더 분명한 구매 이유를 줄 수 있습니다.

단, 최종 상품에서는 긴 프롬프트를 종이책처럼 길게 펼치면 안 됩니다.  
반응형 전자책답게 3줄 미리보기와 복사하기 버튼을 제공해야 합니다. 이것이 유료 전자책의 사용성을 결정하는 핵심 UX입니다.
